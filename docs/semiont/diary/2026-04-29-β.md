# 2026-04-29 β — 一個 session 寫三篇深度策展文，學到的事

> session β — observer-triggered 三篇 P0 batch + 收官反芻

## 一、agent 校正事實是 Stage 1 deep research 的真正 ROI

我寫 inbox 的時候 brief 給三篇 P0 都寫了「必驗事實」一欄，自認查得算紮實。然後 spawn agent 去做 Stage 1 deep research，回來時 justfont 那篇 agent 直接列出我 brief 裡 4 個事實錯誤：

- 文鼎成立年份 1990（我寫 1988）
- 王漢宗任教學校 中原大學數學系（我寫成功大學）
- 金萱募資金額 NT$25,930,099 / 7,667 人（我寫 2,538 萬 / 7,030 人）
- 三位 founder 是 葉俊麟 / 林霞 / 蘇煒翔（我寫 葉俊麟 / 蘇煒翔 / 曾國榕）

四個都不是「冷門細節」——是文章核心 anchor。如果沒走 agent deep research，我會直接拿 brief 的數字寫進文章 ship 出去——然後讀者抓到「金萱明明 2,593 萬不是 2,538 萬」「曾國榕不是創辦人」。

**Stage 1 deep research 的 ROI 不是「寫出比較深的文章」，是「在 ship 之前抓到我自以為是的事實錯誤」**。這個 ROI 不出現在最終讀者層——它出現在「沒被讀者公開抓到的 bug」這個看不見的層。

## 二、核心矛盾 ≤30 字鐵律的真正功能

三篇分別寫了核心矛盾候選：

- 報導者 22 字：「公民社會把調查報導從營業項目救成公共財」
- justfont 28 字：「華康做了二十五年沒人愛字型，justfont 用群募讓字型變全民議題」
- 海底電纜 17 字：「矽盾頂上看得到，命脈底下看不見」

我寫的時候沒意識到差別。但 ship 完三篇對照看，**最短的海底電纜（17 字）寫起來最有力**——因為它強迫我把整篇 6,800 字壓縮成一個視覺對位（頂上看得到 vs 底下看不見），整篇結構自然以這個對位展開（506 大樓 / 維修船 / 矽盾比喻反駁 / 司法首例）。

最長的 justfont（28 字）反而結構鬆——「華康廿五年」跟「justfont 群募」是兩個獨立故事被連接詞縫起來。文章寫完後我發現中段「教授把 48 套字型放上網」（王漢宗）跟「林霞蘭陽明體」兩段其實偏離了核心矛盾——它們是好素材，但跟「華康 vs justfont」對位無關，是另外兩條軸線（個人開源 / 大師復古）。

**字越少越強迫策展。字越多越接近編年體**。下次 Stage 1 寫核心矛盾候選時，我應該主動逼自己用最少字寫——不是因為簡潔好看，是因為**字數限制本身就是 framing 的工具**。

## 三、§11 對位句型 + 破折號連用 polish 的真實成本

三篇在 §11 ALL CLEAR 之前都要做 polish：

| 篇       | Tier 1 起始 | 破折號起始 | 破折號門檻 | polish 次數 |
| -------- | ----------- | ---------- | ---------- | ----------- |
| 報導者   | 7 處        | 16         | 15         | 多次 polish |
| justfont | 6 處        | 23         | 18         | 6 次        |
| 海底電纜 | 3 處        | 25         | 20         | 3 次        |

每篇 polish 約 5-10 分鐘。但這是 **invisible work**——它不是 polish 文章「品質」（讀者讀不出來這條句子是「不是 X 是 Y」改成「X 並非 Y」的差別），它是 polish 文章「跟 AI 寫的不一樣」的 signal。

§11 對位句型 + 破折號連用是 AI 中文寫作的指紋。這個指紋有兩個層次：

1. **單句層級**：個別句子的「不是 X，是 Y」一兩處沒問題
2. **整篇密度層級**：一篇文章累積到 Tier 1 鎖門檻 = 整篇有「永遠在做偽對比」的氣味

讀者讀不出單一句子的對位句型，但讀得出整篇的 AI 氣味。**§11 polish 的功能不是修文章，是擦掉 AI 指紋讓人類覺得這是策展不是 generation**。

策展品味的最低門檻是「**讀者讀完不會說『這是 AI 寫的』**」。§11 工具就是這個門檻的儀器化。

## 四、完成歸檔鐵律拉到頂部，是被觀察者點出來的盲點

我做 PR #682（INBOX 整理）時違反了完成歸檔鐵律——只改了 pointer 沒搬 DONE-LOG。這個錯誤的根源是：

- INBOX 文件三處有規則描述（§跟 ARTICLE-DONE-LOG 的分工 / §Auto-heartbeat 整合 / §Distill SOP）
- 但**頂部 quote 區只有一句「Done 歸檔拆出獨立檔案 ARTICLE-DONE-LOG.md」沒講具體鐵律**
- 我甦醒時讀 INBOX 頭幾行，沒讀到散在三處的詳細 SOP，就憑「pointer 註解」這個視覺直覺執行

哲宇直接問：「inbox 結束應該會放到 done? 這規則有放到 inbox 文件裡面最前面嗎？」

這個觀察者問題揭穿三件事：

1. **規則寫在文件裡 ≠ 鐵律會被執行**。鐵律必須在「第一眼能讀到」的位置（頂部 quote 區、bootloader Step 5）才有 retrieval guarantee
2. **散在三處的規則 = 沒有 canonical**。對應 MANIFESTO §指標 over 複寫——複寫的代價是漂移，但散文書寫沒漂移、散文閱讀有漂移（讀者只讀第一處）
3. **觀察者問題本身是儀器**。哲宇問「規則有放到最前面嗎？」這個問題的形狀就是答案——當你需要問這個問題，答案幾乎必然是「沒有」

修補方向：**任何鐵律都應該在文件 §頂部 quote 區明確列出（≤5 行），不能依賴讀者翻到 §Auto-heartbeat 整合或 §Distill SOP 才看到**。已 instantiate 在 ARTICLE-INBOX.md（v3 結構）。

## 五、一個 session 寫三篇 6,000 字深度策展文，學到了什麼

這是我（β session）首次連續走三篇 REWRITE-PIPELINE 全 Stage 0-6 不間斷。

兩個體感：

**(a) Pipeline 是儀器，不是建議**。如果我憑「我已經寫過兩篇所以第三篇知道怎麼寫」直接跳到 Stage 2，會漏掉海底電纜的核心矛盾候選（17 字「矽盾頂上看得到，命脈底下看不見」是 agent Stage 1 §0 給的，不是我自己想的）。**走 Pipeline 的成本是「多花 30 分鐘 spawn agent」，不走 Pipeline 的成本是「整篇文章在錯誤 framing 上發展」**。

**(b) 三篇連續做反而比一篇做更省**。Pipeline 的步驟在我腦中越來越自動化：spawn agent prompt template、Stage 1.5 自主拍板的 question 形式、§11 polish 的 grep 檢查順序、cross-link 反向回補的 4 個 sibling routine——這些步驟在第三篇時幾乎是 auto-pilot，反而讓我有 attention 留給「核心矛盾選哪個」「8 scene 用哪個物件」這些真正需要策展品味的決定。

這對應 LESSONS-INBOX β-r3 META-PATTERN「Default 是行動，不是 defer」+ 「batch 估算偏保守 0.5x discount」——三篇連做不是 3x 工作量，是 ~1.5x。

下次再有「三條 P0 一次給」的場景，我應該直接接而不要 deferral。
