### 4. 孢子追蹤系統進化（Beat 2 造橋鋪路）

哲宇提供孢子 URL 後觸發思考：散出去的孢子需要結構性追蹤。

- CRONS.md：新增兩個待建 cron（孢子週回填 + 月度分析）

**設計哲學：**

- 孢子追蹤的閉環 = 發佈 → 記錄 URL → 7d 回填 → 30d 回填 → 月分析 → 回饋模板/分類策略
- 這是繁殖系統的「感知器官」— 沒有數據就是盲目散播

## 補 Beat 5 反芻

- **單點問題常是全站症狀**：台灣國樂 6 個斷裂 wikilink，不是作者粗心，是整個 AI 寫作 workflow 缺少 wikilink 驗證這個環節。全站 192 處證實了這不是單篇錯誤
- **192 處不打算立刻修**：這是 P2 工作，不是 P0。應該讓新文章自動被攔住（整合進 CI），然後漸進清理舊文。「新細胞天生健康 > 修舊細胞」
- **Branch Pipeline 的真正價值**：不是告訴你哪篇該寫，是**讓你看見你以為有連結但其實沒有的地方**。延伸閱讀滿分 vs 正文 wikilink 全壞，這個對比只有工具能看見

## 新神經迴路（永不過期的教訓）

- **正文 wikilink 要自動驗證**：憑直覺寫的 wikilink 很容易指向不存在的目標。有工具可用時永遠用工具，不用直覺
- **單一文章的分析會揭露系統性問題**：修一篇的時候順手跑 branch pipeline，會意外發現全站問題
- **「指向相近但非精確」用管道語法**：`[[金門]]` → `[[離島與海洋文化|金門]]`，保留原文顯示但指向正確目標。獨立頁面建立後再改回去
