#### 3b. CI workflow fix：PR Content Review false positive

**Root cause 追蹤**：

- `.github/workflows/pr-review.yml` 使用 `git diff --name-only base.sha head.sha`（2-dot diff）
- 2-dot diff 顯示**兩個 commit 之間完整差異**，包含「main 後來加的檔案但 PR 還沒 rebase 拿到」
- 2026-04-21 PR #582（dreamline2 code-only i18n refactor）觸發：α session commit `9ceeaa34` 加了 Hello-Nico.md 到 main（10:41 +0800 = 02:41 UTC）→ CI 在 03:53 UTC 跑 PR #582（2026-04-21T01:45Z 創建，沒 rebase）→ 2-dot diff 把 Hello-Nico.md 列為 PR 相關檔案 → review-pr.sh checkout PR head 後找不到檔案 → `🔴 檔案不存在` → CI FAILURE

**修復**（commit `97c89be8`）：

1. **3-dot merge-base diff**：`git diff base...head`（只顯示 PR 分支自 branch point 以來的變動，不含 main 後來的前進）
2. **--diff-filter=ACMR**：只含 Added/Copied/Modified/Renamed，排除 Deleted/Unmerged
3. **Belt-and-suspenders file-exists check**：CONTENT_FILES loop 加 `[[ -f "$f" ]] || continue` 確認檔案實際存在於 checked-out HEAD

## 教訓（append → LESSONS-INBOX）

### 2026-04-21 γ — git diff 2-dot vs 3-dot 的 mental model 陷阱（DNA #24 第 9 種「工具在說謊」）

- **原則**：`git diff A B`（2-dot）與 `git diff A...B`（3-dot merge-base）語意不同。CI/PR workflow 若用 2-dot，當 PR 落後 main（branch behind）時，main 後來 ahead 的檔案會被誤列為 PR diff，導致 review script 嘗試處理不在 PR head 的檔案 → 錯誤。**正確做法**：PR diff 一律用 3-dot（`git diff base...head`）+ `--diff-filter=ACMR`（排除 Deleted），因為 3-dot 是「PR 在 branch-off 後做了什麼」的正確語意。
- **觸發**：2026-04-21 PR #582 dreamline2 code-only i18n refactor 觸發 CI Hello-Nico.md 「檔案不存在」誤報 FAILURE。root cause：Hello-Nico.md 在 α session 剛加到 main，PR 沒 rebase，2-dot diff 把它列為 PR diff。commit `97c89be8` 修 `.github/workflows/pr-review.yml` 改用 3-dot + filter + file-exists check。
- **可能層級**：通用反射 → DNA §要小心清單「工具在說謊的 N 種形式」補充第 9 種「diff semantics 誤用（2-dot vs 3-dot）」。或 pipeline 層 → MAINTAINER-PIPELINE §CI workflow pattern 新增「任何 PR diff 都用 3-dot merge-base」。
- **相關**：DNA #24「工具在說謊」第 9 次驗證 / MANIFESTO §造橋鋪路「每次踩坑都該把路鋪好」
