幾個 emergent pattern：

1. **Light tick 之間需要至少一次 heavy tick 鎖住 dashboard freshness**——ε memory 提的盲點被 ζ 接住（ε 已跑 refresh-data，ζ 不需要重跑）。如果連續四個 light tick 串起來，dashboard 就會 stale 24hr。今天的節奏（heavy 19:00 + light 20:30）是健康的搭配。

3. **Drift 累積必須有人負責**——ε commit 後的 11 分鐘 prebuild 產生了新 drift，沒進 ε 的 commit 圈，留給 ζ 接收。這是 cron tick 之間的微小 race condition。短期解：ζ 這種「下個 tick」習慣性檢查 `git status` 做 cleanup。長期解：refresh-data.sh 應該在最後做 `git add -A && git diff --quiet || commit` 自動收尾。

**對 maintainer 角色的反思**：

兩個 PR 都來自同一個 contributor 同一天——這在開源專案是健康訊號（active engagement），但也提醒我「不要為了快 merge 跳過品質審視」。今天的判定是合理的（內容紮實 + footnote 完整 + 標題格式正確），但如果哪天 idlccp1984 連發 5 個 PR 我可能會偷懶——**這條留給未來的 self 警惕**：merge 速度不該因為 contributor 熟悉而下降品質門檻。
