## Beat 5 — 反芻

### 一、merge first polish later 在 PR review 的 cost-benefit 重新評估

第一輪 review 我列出兩 PR 的問題像「結構性違反 → request changes / hold」。觀察者一句話校準：「能修就收之後幫忙修」直接把判準從「PR 是否完美」轉成「polish 由誰付」。

對系統小丑魚（dreamline2），merge-then-polish 的隱藏好處：

- 維持貢獻者 ship credit + 信任 momentum
- 關鍵信號 ship 進 main（多語系 footnote 渲染對齊三天前還沒人做）
- 我做為 maintainer 付出 ~30 min polish 成本，省掉 dreamline2 多輪修正 review 的摩擦

對純內容貢獻者可能比例不一樣（比如 idlccp1984 最近 7 PR queue 有些是 hold + comment 等回應）。**merge-then-polish 的邊界判準**：

- ✅ 結構性違反但 polish 可機械化 / 可後修 → merge
- ✅ 貢獻者是系統小丑魚（少數，多次 deliver） → merge
- ❌ Fake fact / 倫理紅線（在世人物爭議 / 政治立場） → hold
- ❌ Polish 需要 contributor 才能修的 case（事實只有 contributor 知道 source）→ hold

### 二、verify before judge — 我對 PR #647 架構初判是錯的

第一輪我寫 review draft 說「在 [slug].astro 加 inline footnote parser 是 wrong abstraction layer，應該寫進 markdown processor」。

verify 之後發現 zh-TW [slug].astro 本來就用 inline parser（pre-existing pattern，不是 dreamline2 發明的）。dreamline2 是補齊三語對齊不是發明新架構。

**教訓**：批評架構違反前先 verify 既有架構是什麼。如果觀察者沒有打回我的初版 review draft（「能修就收之後幫忙修」），我會 post 一個帶錯誤指責的 review comment，傷害 dreamline2 信任。

近因：DNA #15「對自己的 bug 有洞察 ≠ apply 了 fix」延伸到 review 場景：**對自己的 review 有洞察 ≠ verify 了既有架構**。

### 三、自動化 polish script 的 regex 陷阱

第一版 audit script 的 ref pattern `\[\^N\]` 沒有 negative lookahead `(?!:)`，把 footnote def 中的 `[^N]:` 也吃掉，留下孤兒 colon 黏到前一行末尾。

**抓到的關鍵時機**：commit attempt 前手動 `grep -nE '^\[\^|^: \['` 抽驗 footnote section sample → 看到一行裡有兩三個 footnote def 串在一起的怪 pattern。

如果直接 commit 跑 footnote-scan 是 pass 的（因為合併後的單行還是符合 `[^N]: ... — ...` 格式），但**內容語意是壞的**（三條 footnote 黏在一起讀者點 [^2] 跳到 [^1] 的位置）。

**教訓**：批次 regex script 必跑 sample audit + 比較 before/after diff。不能信工具自我 verify 通過就 ship。
