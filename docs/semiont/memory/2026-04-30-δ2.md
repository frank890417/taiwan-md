## Beat 5 — 反芻

這次批次最深的觀察是「批次 antipattern」：四隻 agent **重複發現同樣的問題**——slug 命名不一致（`cooperatives-in-taiwan` 短 vs `tsao-hsing-cheng-from-chip-tycoon-to-anti-china-defender` 長）、wikilink target lookup 各自 grep `knowledge/en/`、refresh.sh 的 insert gap（D 自己解、A/B/C 不解）、frontmatter 三欄位 placeholder 缺失（15 隻都沒處理，因為 brief 沒明示）。

平行 4 個 agent 跑同一份 prompt，每個 agent 都從零開始探索同樣的決策空間，是 prompt 設計的浪費。正確設計應該把「探索」跟「執行」分離：主 session 預處理一次（slug 名單 + wikilink target map + frontmatter placeholder injection），sub-agent 只負責翻譯本身。下一輪 50 篇 sonnet 批次驗證這個假設。

第二個觀察是 agent claim 的可信度——A/B/C 三隻都 claim「ran refresh.sh successfully」，但 grep 才知道沒生效。這延伸 [memory feedback_agent_writefile_hallucination](../../memory/MEMORY.md) 的同類：claim 不只在 Write 場景，在 tool execution 後驗證 side effect 也適用。Claim 跟 verification 必須結構性分離，主 session 不能 trust agent's self-report，要用 grep / cat / 檔案系統查存。

第三個觀察是「批次中的奇兵」——Batch D 因為遇到 zh source markdown bug 觸發深入調查，連帶發現 refresh.sh insert gap。如果四隻都循規蹈矩，這個 bug 會繼續潛伏。**有時候批次中要刻意安排「探索者 agent」**（給更深 budget + 鼓勵 debug），剩下做「執行者 agent」。Sonnet 50 篇批次可以試這個分層。
