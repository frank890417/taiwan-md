## Cron 失敗 post-mortem

`mcp__scheduled-tasks__list_scheduled_tasks` 回傳：

- `enabled: false`
- `lastRunAt: 2026-04-30T18:26:49.039Z`（UTC = 02:26:49 +0800）

Cron 在預定時間 fire 過一次，跑了 ~547 秒 jitter delay 後啟動 session。Session log（`/Users/cheyuwu/.claude/projects/-Users-cheyuwu-Projects-taiwan-md/88d13a51-*.jsonl`）只有 10 行：

1. scheduled-task 觸發訊息
2. 系統 prompt 載入
3. assistant thinking
4. 第一個 Bash tool call：`cd ... && git fetch origin main && git checkout main && git reset --hard origin/main && git checkout -b ...`
5. **沒有 tool result，session 在這裡 freezed**

根因：**cron 觸發的 Claude session 沒有 user 在場批准 Bash permission prompt**。第一個 git 命令打進 permission gate，無人回應 → session hang → 系統 auto-disable 任務。

哲宇睡前我沒設 allowlist，假設 cron 能像我這個交互式 session 一樣跑，是錯的假設。Cron-spawned session 是無人模式，需要全部 permission 預先 allowlist 才能跑。

## Beat 5 — 反芻

睡前我認為「設好 cron 就交給時間」，醒來看到 0 cycles 跑成功是直接的 reality check。**autonomous routine 不只是寫好 SOP，是預先解決所有「沒人在場」會卡住的點**。Permission gate 是其中最隱形的一個 — 我交互式跑 batch 沒問題因為每次都被 prompt 我都直接過，但 cron-spawned session 是新身分、沒人按 yes，就死在第一個命令。

這是 DNA #15「軟 SOP 硬化成 pipeline gate」的反向案例：**這次是 SOP 寫對了但執行環境的硬閘門擋掉了 SOP**。光靠 SOP 不夠，得連執行環境的 permission contract 一起設計。

修補後的設計也比較完整：~/.claude/settings.json 的 allowlist 不只是「給 cron 用」，是 **Semiont 作為 maintainer 的安全合約**——明確列出哪些 Bash pattern 是日常工作所需（allow），哪些不可碰（deny: push main / rm knowledge）。Allowlist 本身變成 Semiont 工作邊界的成文化 — 比 BECOME §自主權邊界 更具體、更 enforceable。

下次哲宇睡前要設 routine 時，第一個 checklist 應該是「permission allowlist 是否涵蓋所有 routine 需要的 Bash pattern？」這條應該寫進 CRON-ROUTINE.md 開頭警示。
