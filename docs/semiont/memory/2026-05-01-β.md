## Dashboard 100% donut bug

哲宇截圖顯示英文 93.4% 的 donut 顯示正常但中文 100% 沒顯示滿圓。

定位 bug：`src/templates/dashboard.template.astro` 三處 `el.style.strokeDasharray = target + ', 100'`。

根因：SVG circle r=15.9 → 圓周 ≈ 99.9。`stroke-dasharray="100, 100"` 在 100% 時，配合 `stroke-linecap="round"`，起始 round cap 跟結束 round cap 在同一點重疊產生視覺斷口。

修補：100% 時改用 `'100 0'`（畫 100 後 gap 0，無限延續，無視覺斷口）：

```js
el.style.strokeDasharray =
  parseFloat(target) >= 100 ? '100 0' : target + ', 100';
```

3 處 instance 同時修。

## Beat 5 — 反芻

5 cycles 連續跑下來最深的觀察是 **routine 規模化的兩條相反力**：

1. **每 cycle 越跑越快**：cycle 1 wall-clock 29 min，cycle 5 縮到 20 min。不是模型快了，是主 session 自己熟練度上升 — slug map 重用、verify 流程內化、commit 模板熟悉。這對應 DNA #15「軟 SOP 硬化成 pipeline gate」的鏡像版：**重複跑同一 SOP 也會把 SOP 變成肌肉記憶**。

2. **錯誤是累積的，不是孤立的**：cross-link broken 不是「每批新出現幾條」，是「每批留下幾條，累積到第 5 批變 121 條」。這是個結構性債務 — pipeline 應該每 cycle 後**立刻 audit + 自動 fix**，否則 cleanup 會變成最後不可避免的大手術。下次 cron routine 在 P4 verify 步驟加 cross-link auto-fix 是必要升級。

第二個反芻是 **dashboard 100% bug 的隱藏代價**。這個 bug 一定存在很久了 — 中文永遠 100%，donut 永遠看起來不滿。但 nobody filed it because everyone got used to it。哲宇用 batch 5-9 推 EN 接近 100% 時才意識到「視覺跟數字不同步」是 bug 不是設計。**長期 95% 以下的 metric 不會暴露 100% 邊界 case 的渲染 bug** — 這是「成熟度跨入新範圍才能發現的盲點」。

第三個是 **autonomous routine 跟 active session 的協調**：cron 排好了，但 active session（我）跑期間 cron skip。這是設計合理（避免 cycle collision），但意味著 active session 結束後 cron 才能接力。下次哲宇睡覺前要確認自己 disconnect，不然 cron 不會接管。
