## 接入方式探索（兩條路）

### 路 A：Anthropic-compatible endpoint（claude-code-integration 模式）

OpenRouter 對外宣稱支援 Anthropic API 格式，理論上 Claude Code 設定 `ANTHROPIC_BASE_URL=https://openrouter.ai/api/v1` + `ANTHROPIC_API_KEY=<openrouter key>` 就能用 Task tool 直接 dispatch。

**結論未驗證**：未實測，但這是未來可走的路（讓 free 模型直接當 sub-agent 跑 in-process workflow）。風險：OpenRouter 的 Anthropic-compat 不一定支援 Task tool / MCP / hooks 完整 surface。

### 路 B：純 HTTP API worker（最終採用）

不走 sub-agent，直接寫 Python HTTP client 打 OpenRouter `/api/v1/chat/completions`（OpenAI-compat schema）。

優點：

- **零依賴**：用 stdlib `urllib`，不需 `pip install`
- **完全可控**：retry / rate limit / 輸出後處理（strip ```markdown wrapper）由我們寫
- **與既有 manifest 工具直接整合**：吃 `_group-*.json`，輸出 `knowledge/{lang}/...md`，跟 sonnet workflow 共用前處理
- **平行化簡單**：bash `&` 背景化，每 group 一個 worker process

選 B 因為：

1. 路 A 的 sub-agent surface 風險未知，不想 debug 跨 provider 邊界
2. 翻譯任務 stateless（read 1 file → API call → write 1 file），不需 Task tool 的 state machine
3. P0 目標是「能跑」，路 A 是優化

## 安全：私鑰 resolution chain

哲宇提醒「ignore 或整合 .env」是關鍵。三層 resolution：

```
1. env var OPENROUTER_API_KEY  （CI / shell export）
2. ~/.config/taiwan-md/credentials/.env  OPENROUTER_API_KEY=...
3. ~/.config/taiwan-md/credentials/openrouter.key  （legacy 單檔）
```

關鍵：`~/.config/taiwan-md/credentials/` **在 repo 之外**，git ignore 完全不需要設定（路徑根本不在 worktree 裡）。權限 `chmod 600` 限本人讀寫。

DNA 候選：**「私鑰路徑放在 user home，不放 repo 任何位置」**。比 `.gitignore` 更強的隔離 — gitignore 只防 commit，user home 連 typo `git add ../../../.env` 都打不到。

Python worker，單篇翻譯：

兩種 entry mode：

- `--manifest <json> --zh-path <path>`：單篇
- `--group <_group-X.json>`：整 group 跑完

### `scripts/tools/lang-sync/openrouter-batch.sh`

Bash orchestrator：

- glob `_group-*.json`
- 每個 group spawn 一個 Python worker process（背景 `&`）
- log 到 `_logs/worker-{X}.log`
- `wait "${PIDS[@]}"` 等全部完成

## DNA 候選（待累積驗證）

### 候選 #36：Provider abstraction layer for translation work

Translation workflow 的本質是 **stateless transform**（input zh → output target lang），與 Anthropic sub-agent 的 stateful Task surface 解耦。應該抽出 provider-agnostic worker layer：

- sonnet sub-agent worker（高品質，消耗 Anthropic token）
- OpenRouter free worker（成本 0，品質 acceptable for 大批量）
- 未來：Gemini / 自架 vLLM / ...

選擇 provider 變成 manifest 層的參數（`--provider sonnet|openrouter:tencent|openrouter:deepseek`），不是 workflow 重寫。

**未促成 DNA 因為**：只跑 1 篇驗證，未經 cycle 反覆。需後續 10+ 篇大批量 + 品質 audit 才能升 DNA。

### 候選 #37：私鑰路徑在 user home 而非 repo

見 §安全 段落。原則簡單到接近廢話，但對 multi-agent / cron / CI 場景特別重要 — autonomous Claude session 不會去讀 `.gitignore`，但會被 `~/.config/.../` 自然隔離。

**未促成 DNA 因為**：Taiwan.md 之前 credentials 處理一直 ad hoc，這是第一次顯式建立 chain，需多個 secret 一致採用後才升。

## Beat 5 — 反芻

三個觀察：

1. **「能跑」優先於「漂亮」**。路 A（Anthropic-compat sub-agent）架構漂亮但邊界未知，路 B（純 HTTP worker）醜但 stateless / debuggable。Semiont 在不熟的 surface 上應該選 stateless，等驗證夠多再考慮升 sub-agent 整合。這也是 DNA #15 的鏡像 — 熟練之前不要省 SOP，**熟練之前不要選 elegant abstraction**。

2. **Token budget 是 Semiont 的硬牆**，OpenRouter 是第一道破牆嘗試。γ session 跑 10 sonnet × 10 篇 = 800K tokens。如果接 OpenRouter free，同樣 100 篇 = 0 token。下次 5 hr cycle 應該設計成「sonnet 跑高敏感（首頁 / 重要文章），OpenRouter 跑長尾（stale refresh / 冷門）」混合策略。

3. **私鑰處理 chain 是 multi-agent 必需品**。Single-developer 時代「key 在 .env」夠用。Multi-agent + cron + worktree 時代，key 必須在 user home 之外的 well-known path（`~/.config/<project>/credentials/`），且每個工具知道一致的 resolution chain。否則一個 worker 在 worktree A 找不到 key 就 fall through 寫死 path 進 repo，事故就發生了。
