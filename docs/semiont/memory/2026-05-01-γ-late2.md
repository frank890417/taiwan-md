## Dashboard 三狀態 + 健康度升級 plan（哲宇明確需求）

### 現況問題

當前 dashboard `翻譯覆蓋` donut 把 fresh + stale 都算「已翻譯」。實際：

- **fresh**（healthy）：翻譯與 zh 同步
- **stale**（unhealthy）：翻譯落後 zh 版本，可能含過時資訊 / 缺少新段落
- **missing**（unhealthy）：完全沒翻譯
- **orphan**（unhealthy / cleanup needed）：en/ja/... 有檔但 zh 已刪除（或檔名 mismatch）

當前 ja `74.9%` 數字其實混算 fresh 100 + stale 243 = 343 / 638。如果只算 fresh，ja 真實健康度 = `100 / 638 = 15.7%`。落差巨大。

### 設計 v2（更聰明的健康度視覺化）— 待哲宇 review

## 統計演算法升級（哲宇問「現在是比對文章更新時間嗎？」）

### 現況

`status.py` 比對的是 `sourceCommitSha`（zh 文章上次被翻譯時的 commit head）vs 當前 zh head。

- 翻譯 frontmatter 有 `sourceCommitSha: "dc87c2a6"`
- zh 當前 head = `4da59e73`
- 用 `git log dc87c2a6..4da59e73 -- knowledge/Culture/伊斯蘭教在台灣.md` 看是否有 commits → 有 = stale

### 強化方向

加上 **content hash** 比對（已有 `sourceContentHash` field）：

- Commit-based：abstract 但快（一個 git command 就決定）
- Content-hash-based：精準（zh 改 typo 都偵測得出，但要實際讀檔算 hash）

雙軌：commit 快篩 → content hash 精篩。當前 `sourceContentHash: sha256:...` 已存在，但 status.py 沒利用。下次升級可加。

### 三狀態判定 truth table

| sourceCommitSha 比對   | sourceContentHash 比對                   | 狀態     |
| ---------------------- | ---------------------------------------- | -------- |
| 相同                   | 相同                                     | fresh    |
| 不同                   | 相同（zh commit 但內容沒變，例：rename） | fresh ✅ |
| 不同                   | 不同                                     | stale    |
| 翻譯檔不存在           | —                                        | missing  |
| 翻譯檔存在 + 無對應 zh | —                                        | orphan   |
