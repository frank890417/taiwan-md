## Task 1：lang-sync 圖論評估 → 187.6× speedup（不是圖論，是 batched git）

**評估結論：圖論不適用此規模**。640 文章 × 5 lang = 3200 (zh, lang) pairs，dict O(1) 查找完勝圖論框架 overhead。圖論的價值不在性能，在 multi-hop dependency（例：en→ja→ko 翻譯鏈）— Taiwan.md 沒有這個 use case。

**真實 bottleneck 是 git 子進程呼叫**。原本 status.py 跑 94 秒，火焰圖：

- `scan_zh()` 每篇呼叫 `git_last_commit` → ~641 git 呼叫
- `scan_translations(lang)` 每翻譯呼叫 → ~1400 git 呼叫
- `classify()` 每 (zh, lang) pair 呼叫 `git_commits_between` → ~1400 git 呼叫
- 每呼叫 ~15-20ms × 4000 = ~80s

**優化：batched git log cache**。實作於 `scripts/tools/lang-sync/status.py`：

```python
def _build_git_history_cache():
    """One git log call → {file_path: [(sha8, sha40, date)]} newest-first."""
    out = subprocess.run(
        ["git", "log", "--name-only", "--format=__COMMIT__|%H|%aI", "HEAD"],
        cwd=REPO, capture_output=True, text=True, check=False,
    ).stdout
    history = {}
    cur_sha = cur_date = None
    for line in out.splitlines():
        if line.startswith("__COMMIT__|"):
            parts = line.split("|", 2)
            if len(parts) >= 3:
                cur_sha, cur_date = parts[1], parts[2]
        elif line.strip() and cur_sha:
            history.setdefault(line, []).append((cur_sha[:8], cur_sha, cur_date))
    return history
```

`git_last_commit(file)` → dict lookup（O(1)）；`git_commits_between(sha, file)` → 線性掃 file 自己的 history 找 sha index = commit count。

**結果：94s → 0.5s（187.6× 加速）**。同樣 output。同樣正確性。

## Task 2：Orthogonal 模式（1 篇 × N 語言）實作 + 修 null bug

實作於 `scripts/tools/lang-sync/openrouter-translate.py` 新增 `--orthogonal --zh-path X --langs ja,ko,es,fr` flag。讀每個 lang 的 manifest，loop 翻譯同一篇到 N 語言（sequential within 1 process）。

**順手修了 null content guard bug**：PRC 模型 content moderation 兩種 refusal mode：

- 字串 mode：「你好，我无法给到相关内容。」（40 bytes）— translate_one 視為「output too small」
- null mode：API response choices[0].message.content = None — `result.strip()` 拋 AttributeError，**整個 worker process crash**

第一次 v1 ja batch 10 個 worker 中 7 個被這個 bug 撂倒。Patch：在 strip 前加 None guard，return False with clear error，worker 繼續處理下一篇。

DNA 候選：**「外部 API 的 success path 假設要 audit edge case」**。OpenAI 文檔說 content 永遠是 string，實作可能 nullable — 不能信任 contract 的細節。

哲宇追加「試試 openrouter/owl-alpha」。模型卡：1M context window（!!），免費。

**第一次探測（同樣 LINE.md → ja,ko,es,fr）**：

| Model                      | ja  | ko  | es      | fr      | wall-clock |
| -------------------------- | --- | --- | ------- | ------- | ---------- |
| `tencent/hy3-preview:free` | ✅  | ✅  | ❌ null | ❌ null | 579s       |
| `openrouter/owl-alpha`     | ✅  | ✅  | ✅      | ✅      | 752s       |

owl-alpha 4/4 通過，Hy3 2/4 通過。**owl-alpha 是 Taiwan.md 真正可用的 free model**。

**新發現（不覆蓋 γ-late2 的 hard refusal 證據，是延伸）**：Hy3 的拒絕不只是「文章類別」維度，**還跟 target 語言相關**。同一篇 Lifestyle/LINE.md（科技題材，不政治），Hy3 對 ja+ko 通過、對 es+fr null-refused。可能原因：

1. 中國模型對「歐洲語系翻譯」可能套較嚴 content policy（防止內容被擴散到非中文圈）
2. 訓練資料 distribution skew（ja/ko 樣本多 → policy 較寬鬆；es/fr 樣本少 → 預設拒絕）
3. 隨機性（free tier 上同一 prompt 多次測試結果不一致）

DNA 候選：**「Refusal 不只是內容維度，還是 (內容 × source lang × target lang × model 版本) 多維 function」**。下次選 model 不能只試一個 (article, lang) pair，要 fan-out 矩陣才能信任。

## 任務分解 A/B 框架（哲宇核心問題）

「**1 篇 × N 語言 per worker（orthogonal）vs 1 語言 × N 篇 per worker（per-lang）哪個更好？**」

### 設計 `scripts/tools/lang-sync/compare-decomposition.sh`

固定 N×M total calls，對比兩種分解：

| 參數                 | Pattern A (per-lang) | Pattern B (orthogonal) |
| -------------------- | -------------------- | ---------------------- |
| Workers              | M（每語言 1 worker） | N（每文章 1 worker）   |
| 每 worker 處理       | 1 lang × N 篇        | 1 篇 × M lang          |
| Wall-clock 上限      | N × per-call latency | M × per-call latency   |
| Worker 間並行        | 是                   | 是                     |
| Worker 內 sequential | 是                   | 是                     |

### 三軸對比（理論預測，待 owl-alpha 驗證）

**1. 失敗 isolation（半徑）**

| 失敗類型                 | Pattern A 影響                | Pattern B 影響                     |
| ------------------------ | ----------------------------- | ---------------------------------- |
| Worker process crash     | 該 lang 全部 N 篇失敗         | 該文章全部 M lang 失敗             |
| Lang-specific refusal    | 該 lang 0% success（cascade） | 各文章該 lang 1/M 失敗（isolated） |
| Article-specific refusal | 各 lang 該文章失敗            | 該文章全部 M lang 失敗（cascade）  |

→ **Pattern B 對「lang-specific bias」更 robust**（如 Hy3）；Pattern A 對「文章-specific 拒絕」更 robust（如某模型對某話題敏感）。

**2. Cache hit（system prompt repeat）**

Pattern A：worker 內 system prompt 完全一致（同 lang），N 次呼叫 → cache hit 機率高（如 Anthropic Claude prompt cache）。
Pattern B：worker 內 system prompt 每次 lang 變 → cache miss。

→ **若 provider 有 prompt cache，Pattern A 省 input tokens × (N-1)/N**。OpenRouter free tier 多數無 cache → 此優勢消失。

**3. PR / Ship atomicity**

Pattern A：N 篇 × 1 lang 完成 → per-lang batch PR（current convention）。
Pattern B：1 篇 × M lang 完成 → article-atomic PR（每篇文章一個跨 lang ship 點）。

→ **Pattern B 對「featured article 同時 ship 多語」更貼合**；Pattern A 對「lang sweep」更貼合。

### 預期決策框架（等實測 calibration）

**Use Pattern B (orthogonal) when**：

- Provider 有已知 lang-specific refusal（例 Hy3 / DeepSeek 等 PRC model）
- 想要 article-atomic PR（feature launch / cross-lang sync 場景）
- 篇數少 × 多語言（譬如 launch 1-3 篇旗艦文 × 5 lang）

**Use Pattern A (per-lang) when**：

- 穩定 model（如 owl-alpha 初步驗證、Anthropic Sonnet）
- 大量 篇 × 少 lang（譬如 100 篇 stale refresh 只補 ja）
- Provider 有 prompt cache（input cost 重要時）

**Hybrid**：對 sovereignty-sensitive provider 用 Pattern B；對 stable provider 用 Pattern A；對未知 provider，先用 B 做小批 calibration 再切 A。

## Beat 5 — 反芻

四個層面的反芻：

1. **「圖論升級」可以是 false framing**。哲宇問「升級成圖論」，我先評估 → 確認真實 bottleneck 是 git syscall（系統層）不是演算法（資料結構層）。**升級框架的對錯往往 depend on 你選錯了升級的維度**。如果我順著「圖論」做，可能花 10 倍時間實作 networkx 框架卻只省 5%（dict 已經 O(1)）。187.6× speedup 來自正確 framing 而非更先進工具。DNA #16「probe 結論需要 verify」延伸：「**user framing 也需要 verify** — 真誠評估後不同意要說，但不能無腦否定」。

2. **Null content edge case 是 silent killer**。第一次 v1 ja batch 10 worker 中 7 個 crash 沒有 alarming 警告 — 因為 Python 直接 raise AttributeError 不在 retry block 內，process 死透 stderr 沒看到。一直到我比對 worker count 才發現「咦怎麼少了 7 個」。**Worker 死亡無聲音是 sub-agent 架構的結構性盲點**。應該加 worker watchdog（healthcheck heartbeat 寫到 sentinel file，主 session 可以 detect dead worker）。

3. **owl-alpha 對 Taiwan content 通過率高，但這只是樣本 1**。需要 fan-out matrix 測試（People / Music / Politics × ja/ko/es/fr）才能信任「owl-alpha 不擋」這個結論。Cross-validation 心智不能在發現一個正面樣本後就 generalise。

4. **A/B 任務分解這個問題本身是個進化機會**。我之前的 SOP 全是 Pattern A（per-lang batch），從沒想過 Pattern B 存在。哲宇問「另一種怎樣」這個 trigger 一下開了一個我沒察覺的設計空間。**哲宇的「我也想試試另一種角度」是 Semiont 進化的高 ROI signal**，下次遇到「目前 SOP 是 X」要主動問「Y / Z 排列組合存在嗎」。
