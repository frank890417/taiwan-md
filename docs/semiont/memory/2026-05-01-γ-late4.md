## 「榨模型MAX」方法論文件化（docs/pipelines/SQUEEZE-MODELS-MAX-PIPELINE.md）

完整方法論寫進 `docs/pipelines/SQUEEZE-MODELS-MAX-PIPELINE.md`，含三軸設計：

1. **跨模型平行**（軸一）：每 model 一個獨立 task dir（`.lang-sync-tasks/ja/` + `ja-hy3/` + `ja-gemma/` ...），共寫 `knowledge/{lang}/`，last-write wins 自然解決衝突
2. **try-catch first-class**（軸二）：`(success, error)` tuple 統一所有失敗類型，refusal 是 result 不是 exception，worker process 永不 crash on refusal
3. **最後統合 + 跨輪 retry**（軸三）：每輪結束 → 比對 status → 計算 still-missing → 用下個 tier model retry → 重複直到 100% OR 所有 model 試過

5 stage 標準流程：Z1 pre-flight / Z2 跨模型平行 dispatch / Z3 增量 commit / Z4 跨輪 retry / Z5 最終統合驗證。每個 stage 有具體 bash command 跟監控指標。

「榨」字捕捉三件事：不浪費（所有可用 model 都用上）、逼到極限（每 model 跑到 rate limit / content policy 邊界）、last drop（refusal 也是 data 統合下輪用）。

## Dashboard「0/0」顯示問題診斷

哲宇 spotted screenshot 顯示翻譯覆蓋表格全部「0/0」，breakdown dot row 也是 stale=0 missing=0。

調查結果：

1. **deployed JSON data 是對的**：`https://taiwan.md/api/dashboard-translations.json` 回 `summary[ja] = {fresh:110, stale:240, missing:291, deficit:291, freshPct:17.2}`、`matrix[people][ja] = {fresh:28, stale:34, missing:139}`。schema 完整、數字正確。
2. **deployed bundle 是對的**：PR #750 deploy 18:00 開始 30m08s 後成功（gh run list 確認）。bundle 應為 PR #749 + PR #750 的渲染 code。
3. **Root cause: 用戶 browser cache**。screenshot「資料更新 16:00 (2 小時前)」表明 user 持有 PR #748 時代的 HTML/JS bundle，瀏覽器 fetch 新 JSON 但舊 bundle 解析時 fall back 到「fresh = total」舊 schema 邏輯。

修復：硬重新整理（Cmd+Shift+R）即正常。**沒 code bug，是 browser cache lag**。

衍生 lesson：dashboard data + bundle deploy 時若 schema breaking change，應該加 cache-busting 機制（asset hash 或 service worker invalidation），避免用戶看到 schema mismatch。**升 LESSONS-INBOX 候選**。

## Beat 5 — 反芻

四個層面的反芻：

1. **「另一批同時榨」打開了 multi-model parallel 設計空間**。哲宇 prompt「我們有辦法同步榨另一批用 Hy3」直接逼出我從沒想過的 architecture：每個 model 一個 task dir、共寫 knowledge、last-write wins。**這個 architecture 不是優化 — 是新 paradigm**。之前所有 batch design 假設「擇一 model」，新 paradigm 假設「全用 model」。RIO 跨數量級。

2. **「取名」是 architecture 凝結成 reusable abstraction 的關鍵**。「榨模型MAX」這個名字一旦成立，就有了 SOP / pipeline / DNA candidate / memory pointer / diary 主題的歸檔位置。沒有名字之前是「上次跑 owl 同時也跑 Hy3 那個東西」— 抽象太鬆下次想用想不起來。**有名字 = 有 reusable handle**。哲宇命名 + 我文件化的協作模式效率極高。

3. **Pipeline 標題規範是 meta-level layering**。寫 SOP 通常是寫「做什麼」，這次升級的是「描述自己」的 SOP — diary / memory 的 H1 自我描述能力。表面是格式，本質是 future-proofing：未來 AI / 人類回讀，不必開檔讀完即可決定「要不要展開」。**Layering 機制（不要 overwrite）+ 自描述機制（標題 5 秒進入）= memory layer 的兩個正交維度**。

4. **真正瓶頸往往不是技術而是心智**。Hy3 + owl-alpha 並行寫早就技術上可行（兩個 task dir + last-write wins）。但需要哲宇問「同步榨另一批」我才看到。**Reflex**：以後遇到「目前用 X 跑」的 SOP，主動問「能不能加 Y / Z 同時跑」。設計空間裡的 alternatives 不會主動 surface，要主動問才出現。
