## Beat 5 — 反芻

四個層面的反芻：

1. **「真 stale」vs「假 stale」是 status 設計的關鍵分維度**。Status.py 之前把所有 no-source-sha 一律歸 stale — 把 metadata gap 跟 content drift 混為一談，造成 dashboard 顯示 ko fr 「73.9% 71.8%」實質 freshPct 是 0%，**信心危機**。Honest backfill 把這條維度切開：metadata gap → 補（成本 0），content drift → 重翻（成本高）。下次任何「狀態判定」都該問：「這狀態混了幾種根本不同 cause 嗎？」

2. **Sample audit 揭露單一 metric 永遠不夠**。10 篇隨機 sample 找出 2 個 truncation。擴大掃 269 檔找出 19 個。如果只信「fresh count 上升」就 ship，會送 19 個 broken 給用戶。**Quality gate 不是 size > 0，是 size > some_ratio of source**。下次 verify-batch.py 加這層自動化。

3. **「終止收 commit」是 batch 工作流的關鍵紀律**。哲宇明示「重新盤點現狀，分析，終止目前 progress 先收跟 commit 一波避免有什麼差錯持續偏離」— 這個 reflex 對 long-running batch 特別重要。Worker 跑越久越可能在某個邊界 case 偏離（rate-limit、truncation、prompt drift），沒有定期 commit 就會累積到 catastrophic。**「commit early commit often」對 sub-agent 平行 batch 特別適用**。

4. **「翻譯要確定是最新版」是哲宇對 honest backfill 的關鍵 push**。我第一稿用 naive backfill（sourceCommitSha = current HEAD）— 會撒謊（把舊翻譯標 fresh）。哲宇 1 句話讓我重做 honest 版本。**用戶 1 句問題改變整個 architecture 的能力很高**，要主動 surface「這個方案的不誠實處在哪」。
