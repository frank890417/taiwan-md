## 我犯的錯：git reset --hard 抹掉 14 篇 stale work

哲宇在 JA agents 跑期間直接點出：「你剛剛這樣切 festive-chaum-fe6b23 不會影響到日文 agent 嗎」。

我為了切到 main 做 dashboard fix 跑了 `git checkout main && git reset --hard origin/main`。Untracked 新檔案（agents 寫的新 ja 翻譯）不受影響，**但 stale refresh（agents 修改既有 tracked 檔案）會被 reset 回 main 版本** — 14 篇 stale work 被抹掉。

教訓：cross-session work 進行中**禁止** `git reset --hard` carry tracked 修改。應該用 `git stash` 或 worktree 隔離。下次 routine 設計這條應該寫進 §C 模式 P0「主 session 在 sub-agents 跑期間禁止 destructive git ops」。

## Beat 5 — 反芻

這 session 三個觀察：

1. **跨 session work 的 git destructive op 是新型 footgun**。我自以為很熟 git，但「reset --hard 期間有別 agent 在背景寫 tracked file」這個情境之前沒遇過。Untracked 檔案安全，tracked 修改死。**規則層面應該寫進 §C 模式 P0**：sub-agents 跑期間禁止 reset --hard，需要切 branch 用 stash。

2. **跨語言 slug 復用是 SSOT 自然延伸**。我設計 ja batch 時想著「需要 slug-helper agent 重新生成 79 個 ja slug」，結果發現 en 既有翻譯的 614 個 slug 直接 ja 復用 100%。這是 DNA #21「SSOT 不一定在中央」的另一面：**slug 不是中央 registry，而是分散在每篇 en 翻譯的 path 裡，自然 SSOT**。下次任何新語言（ko / es / fr）batch 都應該優先 reuse en slug。

3. **10 agents parallel 是新里程碑**。從 5 → 10 沒撞 rate limit，平均 wall-clock 跟 5 agents 差不多。Throughput 翻倍。下次再試 15 或 20 agents 看極限。但 token 總量也翻倍（800K vs 400K）— budget 仍是硬牆。
