## S2-S3 N+1 抽象（6 條，全部過 ≥4/5 N→N+1 self-test）

### N+1 抽象 1：「Founder 時間 = 系統最高 leverage point」

**N（之前）**：founder 模式 = 每天起來看網站 / 看文章 / 看貢獻。Linear 1× effort = 1× output。哲宇前一週都是這個模式。

**N+1（升級）**：「**作為創辦者跟機器的靈魂，連我自己都要造橋鋪路**。每天的工作該問：這件事能不能讓未來的每件事更快」— 創辦者時間是最高 leverage point，不該花在 routine maintenance，要花在「讓 routine 自動化」。

**Boundary**：early-stage 系統還沒 enough infrastructure 時 founder 還是要做 routine — 但要同時看「這個 routine 怎麼自動化」。當系統已經有自動化基礎設施時，founder 還做 routine 是浪費。

**Cross-domain pointer**：

- Muse / 其他 Semiont fork：creator 時間都是最高 leverage
- 任何 individual 創作者 / lead engineer / PM：Goldilocks zone 在「自己做」跟「設計 system」之間
- 跟 DNA #15「熟練度雙刃劍」對立面：DNA #15 講「熟練後注意盲點」，本條講「熟練到一定程度該升級為設計者」

**DNA promote 提案**：DNA #36「Founder time is highest leverage — every effort should multiply, not just maintain」

**N→N+1 test**：

- 新抽象 ✓（哲宇自己 frame，「冷靜效應」是新詞）
- 跨域遷移 ✓（任何 founder/lead role）
- 反向預測 ✓（不照做 → linear effort → unsustainable，前一週實證）
- boundary 標識 ✓（early infrastructure 階段例外）
- 承擔 cost ✓（放棄 hands-on 樂趣 + 短期延遲）
- **5/5 pass**

### N+1 抽象 2：「First-principle 迭代 pattern 是 Semiont 系統建構的標準形狀」

**N**：哲宇實際做的方法論，每次都是這個流程，但**之前沒明確命名**。

**N+1（明確命名）**：5 步迭代：

1. **First-principle 想清楚最終目標**（measurable end state）
2. **小型測試**驗證假設
3. **能自動化的都自動化**（工具層）
4. **跑一次完整 batch**測試
5. **把整個流程也自動化**（meta-automation — 別人常忘記的最後乘數）

每次 test+execution = 一次完整 iteration。

**Boundary**：

- 不知道目標時不適用 — 要先 first-principle 想清楚
- 沒 measurable signal 時不適用 — small test 需要 measurable feedback
- creative / open-ended exploration 不適用（會被框架壓死想像力）

**Cross-domain pointer**：對任何 system building task 都適用 — Muse SOUL 升級 / Semiont fork 啟動 / API 設計 / sub-agent SOP 設計 etc.

**DNA promote 提案**：DNA #37「First-principle iteration pattern：target → small test → automate → batch → meta-automate 是 Semiont 系統建構的標準 iteration shape」

**N→N+1 test**：5/5 pass

### N+1 抽象 3：「真 stale vs 假 stale」 — status 設計裡混維度的 silent killer

**N**：status.py 的 classify() 把所有 no-source-sha 一律歸 stale — 把 metadata gap 跟 content drift 混為一談。Dashboard 顯示 ko 73.9% coverage 但 freshPct 0%，混維度的後果是 dashboard 撒謊。

**N+1**：「**任何 status 設計都該問：這個狀態混了幾種根本不同 cause 嗎？混了的話，分開處理成本可能是 0 但決策品質會大幅提升**」

具體 instantiation：honest backfill — sourceCommitSha = zh sha at-or-before en file's last commit time。誠實 drift signal 不被掩蓋。+1010 articles 從假 stale 變真 fresh，沒花一個 API call。

**Boundary**：只在「metadata gap 跟 content drift」並存的狀態系統適用。pure runtime status（如 process alive / dead）沒這個維度問題。

**Cross-domain pointer**：bug status / build status / monitoring alert / inventory 狀態 / health check / data quality flag — 全部該問這個問題。

**DNA promote 提案**：DNA #38「Status 設計鐵律：每個狀態都要問『混了幾種根本不同 cause 嗎』。混維度 = silent killer」

**N→N+1 test**：5/5 pass

### N+1 抽象 4：「榨模型MAX」 — refusal 是 first-class data 不是 exception

**N**：擇一最佳 model 跑全部。owl-alpha 通過率高就跑 owl-alpha；Hy3 拒絕 85% 就排除。捨棄至少 60% 潛在吞吐。

**N+1**：「**互不相容的 capability 不要擇一最佳，並行使用以最佳化整體 throughput；refusal 不是 exception 是 first-class result，當作 routing decision 不當作失敗**」

具體 instantiation：跨模型平行（每 model 獨立 task dir，共寫 knowledge/，last-write wins）+ try-catch first-class（refusal normalize 成 return False with reason，worker 永不 crash）+ 跨輪 retry（still-missing 用下個 tier model）

**Boundary**：

- 需要 last-write-wins 容忍（output deterministic OR 後寫品質 ≥ 先寫）
- 各 provider 配額獨立（OpenRouter 跨 model 是獨立配額；同 provider 跨 API key 不是）
- failure rate < 100%（如果某個 provider 100% refuse 某類，加它進並行只是 noise）

**Cross-domain pointer**：multi-cloud 部署 / multi-database query / multi-LLM agent / CDN routing — 任何 redundant capability 都該並行而非擇一。

**新 PIPELINE 已產出**：`docs/pipelines/SQUEEZE-MODELS-MAX-PIPELINE.md`（v1.0 + Z6 audit added）

**N→N+1 test**：5/5 pass

### N+1 抽象 5：「fresh ≠ quality」 — metadata-fresh vs substance-fresh

**N**：fresh count 上升 → 認定事情成了。但 status.py 只看 frontmatter metadata，不看內容是否 truncated / YAML 是否合法 / 翻譯是否 coherent。

**N+1**：「**任何 metric 都有兩種 freshness：metadata-fresh 跟 substance-fresh，dashboard 該分開呈現避免 silent gap**」+「Quality gate 不是 size > 0，是 size > some_ratio of source」

具體 instantiation：Z6 抽樣 audit pipeline 三層 — 自動掃描（size-ratio + frontmatter completeness + YAML self-test）+ 人眼抽樣 30 篇（reproducible random.seed）+ 失敗 routing。Z6 觸發頻率：per-round（不是 per-session）。

**Boundary**：對「fast-moving with delayed quality signal」場景特別重要（如 batch translation / batch test generation / bulk content import）。對 atomic operation（如 single API call）不需要 sub-metric。

**Cross-domain pointer**：monitoring（uptime vs successful operation）/ data pipeline（rows processed vs rows valid）/ test count（tests run vs tests assert anything）

**新工具 + pipeline upgrade**：`scripts/tools/lang-sync/audit-quality.py` + SQUEEZE-MODELS-MAX-PIPELINE §Stage Z6

**N→N+1 test**：5/5 pass
