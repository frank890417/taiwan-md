## Part 2 — Bug 發現 + Root Cause 分析

### Bug #1：Tailwind Phase 6 反向 sed（P0 嚴重）

**Commit**: `a7cffefd` (2026-04-10 19:34)
**影響期間**: 2 天（04-10 19:34 → 04-12 ~10:20 修復）
**影響範圍**: **所有 /ja/_ 和 /ko/_ 文章頁面**（257 JA + 28 KO 翻譯全部孤兒化）

**What happened**:

- Tailwind migration 從 en/[slug].astro 做一次，然後 sed 到 ja/ 和 ko/
- Commit message 說「sed knowledge/en→ja/ko」但 **actual diff 方向反了**
- 結果：ja/ko slug.astro 的 `getStaticPaths` 讀 `knowledge/en/`，服務 EN 內容
- `const lang = 'en'` 也沒改（本來就從 en 複製來的 bug，但 Tailwind migration 沒修）

**Symptoms**:

- 每個 /ja/ 頁面顯示 EN 內容 + `<html lang="en">`
- 語言切換器連結用 EN slug 建，但 `_translations.json` 的 ja 條目用 JA native slug → map lookup miss → fallback 用 EN slug 當 ZH URL → 404
- Smart 404 也偵測不到 JA 翻譯（因為用 EN slug 查 knowledge/ja/ 檔案，但 JA 用不同 slug）

**Fix**: Commit `91d28515` — 重寫 getStaticPaths 用 `_translations.json` 做 en→zh→ja 雙跳查找

### Bug #2：\_translations.json 36 個 stale zh 指標

**影響**: 語言切換器自信地指向不存在的 ZH 檔案
**Root cause**: zh 檔案被 rename/consolidate 但 `_translations.json` mapping 沒更新
**Fix**: Commit `b99a9db3` — 修 20 個、刪 16 個無法修復的、加 defensive filter

### Bug #3：27 個 KO 翻譯未註冊

**影響**: 28 篇 KO 翻譯只有 1 篇在 `_translations.json`，其他 27 篇翻譯人力完全浪費
**Root cause**: 日韓翻譯基礎設施誕生時（`dc6d4e8b`, 2026-04-07）只加了 1 個 ko 條目作為 demo
**Fix**: Commit `b99a9db3` — 全部 27 篇找到正確 zh 對應並註冊

### Bug #4：article-index.json Smart 404 ja/ko 盲目

**影響**: Smart 404 對所有 ja/ko 翻譯說「沒有 ja 版本」（即使有）
**Root cause**: 用 `fs.existsSync(knowledge/ja/<Cat>/<enSlug>.md)` 查 JA 檔案，但 JA 用 native slug
**Fix**: Commit `91d28515` — 改用 `_translations.json` reverse lookup

## Part 4 — 防禦層建設

### 已建的防禦

1. **`getLangSwitchPath.ts` defensive filter** — module-level `getValidZhFiles()` cache，每次 build 只掃一次，過濾所有 stale \_translations.json 條目。即使未來有人 rename zh 檔案忘了更新 mapping，語言切換器不會自信地指向 404。

2. **`generate-api.js` stale filter** — article-index.json 生成前過濾 stale 條目。Smart 404 的 `langs` 陣列只包含實際存在的翻譯。

3. **`scripts/tools/git-session-span.sh`** — 造橋鋪路工具，但跟 i18n 無關。

### 待建的防禦（建議優先序）

#### P0：Post-build internal link verifier

- **工具**：`scripts/tools/verify-internal-links.sh`（正在建造中）
- **功能**：掃描 dist/ 所有 HTML 的內部連結，驗證目標存在，特別檢查語言切換連結
- **用途**：每次 build 後跑，CI 門檻（broken ratio > 1% 就紅燈）
- **如果 Tailwind Phase 6 當天有跑這個工具，bug 不會存在 2 天**

#### P1：AI crawler path 死連結告警

- 結合 `fetch-cloudflare.py` 的 AI crawler data，定期比對：
  - AI crawler 最常爬的路徑列表 vs 實際 build 的路徑
  - 如果 top crawled path 在 build 裡不存在 → 紅燈
- 需要 Cloudflare 付費方案或 dashboard 手動匯出才能拿到 per-path 4xx

#### P2：使用者模擬測試

- 腳本：隨機選 N 篇文章 → 模擬「使用者在 /en/ 頁面切換到中文、日文、韓文」
- 對每個語言切換連結，驗證：(a) 目標頁面 build 存在，(b) 目標頁面的 H1 語言正確
- 同樣可整合進 CI

#### P3：\_translations.json stale 預警

- 每次 sync.sh 或 build 後跑 `_translations.json` stale check
- 如果偵測到新的 stale 條目就 warn（不 block build，但在 console 顯示）

### Broken link 分類

| 類型                                 | 數量 | 主因                                                                                                                |
| ------------------------------------ | ---- | ------------------------------------------------------------------------------------------------------------------- |
| **(a) Language switcher dead links** | 102  | 切換到未翻譯語言時，fallback 用錯 slug 格式（zh slug 當 en URL，或 en slug 當 ja URL）                              |
| **(b) Article internal dead links**  | 70   | markdown 內容中的 cross-reference 指向不存在的 slug（如 `/people/ang-lee`、`/lifestyle/convenience-store-culture`） |
| (c) Other                            | 0    | —                                                                                                                   |

### 語言切換 102 個 broken 的子分類

1. **zh → en 切換**（13 個）：zh 頁面的 "Switch to English" 連結用 zh slug 編碼後當 en URL → `/en/food/台灣鹽酥雞` 不存在（en 版叫 `/en/food/salt-and-pepper-chicken`）。Root cause：`_translations.json` 沒有這些 en 條目。
2. **zh/en → ja 切換**（56 個）：ja 翻譯不存在的文章 → 語言切換器 fallback 給了一個 /ja/ URL 但沒有 ja page 被 build。Smart 404 會接手但 UX 不佳。
3. **ko 全面**（70 個）：ko 只有 28 篇翻譯但 ko Hub 列出所有 en 文章 → 大量 dead click。需要 ko Hub 也做翻譯過濾（同 ja slug 修復方式）。
4. **ko 頁面 → /ko/graph 和 /ko/terminology**（多筆）：這些頁面本身不存在（沒有 ko 版本的 /graph 和 /terminology）。

### 文章 cross-reference 70 個 broken 的特徵

- 主要在 en/ 文章裡的 wikilink 或 markdown 連結
- 連結格式像 `/people/ang-lee`、`/lifestyle/convenience-store-culture` — 用 en slug 格式但指向 zh 路由（zh 路由用中文 slug）
- 這是內容層 bug 而非 i18n 基礎設施 bug

## Part 6 — DNA 反射候選

### 候選 #19：大型 refactor 後必須 visual smoke test 多語言 slug 頁面

觸發事件：Tailwind Phase 6 (`a7cffefd`) commit message 說「sed knowledge/en→ja/ko」但 actual diff 反方向。2 天期間所有 /ja/_ 和 /ko/_ 服務 EN 內容。

**操作性建議**：

1. 任何 sed/批次替換 commit 後，`git diff` 確認替換方向正確
2. build 後，至少打開 3 個 URL：一個 /ja/、一個 /ko/、一個 /en/ → 確認 `<html lang>` 正確 + H1 語言正確
3. 跑 `scripts/tools/verify-internal-links.sh --sample 50` 作為 smoke test
4. **這不是可選的「建議事項」，是硬規則**。因為這類 bug 的代價不只是「使用者體驗不好」，是 **AI crawler 把錯誤內容寫進 LLM 訓練權重**，一旦寫進去要等下一輪訓練才能修。
