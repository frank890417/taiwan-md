## 2. 404 頁面 26% 流量：感知盲點的最深處

> **GA4 告訴我們「404 被訪問 67 次」，但沒告訴我們「哪些 URL 在 404」。這是整個感知系統最暗的盲點。**

### 問題現況

| 指標                          | 值                               |
| ----------------------------- | -------------------------------- |
| 4/10 當日 404 頁面 views      | **67**                           |
| 當日首頁 views                | 258                              |
| 404/首頁比率                  | **26%**                          |
| 404 在 top landing pages 排名 | **第 2 名**                      |
| 讀者體驗                      | 四分之一讀者到門口發現房間不存在 |

### 為什麼無法診斷

檢查 `src/pages/404.astro`：

- ✅ Smart 404 存在（session δ 於 2026-04-08 建立）
- ✅ build-time article index（455 篇），支援跨語言 fallback 建議
- ✅ 4 種 i18n 介面（zh-TW / en / ja / ko）
- ❌ **零 analytics instrumentation**——頁面載入時不發送任何事件，GA4 只能看到「一個 404 頁面被看了」，不知道是哪個原始 URL 觸發的

這是 Smart 404 建立時的盲點：做了使用者體驗的 UI 改善（404 找不到文章時顯示建議），但沒有做 observability（不知道哪些 URL 在 404）。**看得見門口，看不見門外的人在找什麼**。

### 404 的四種類型（假設推論，需資料驗證）

| 類型                   | 來源                                            | Smart 404 能幫忙嗎？         | 對策             |
| ---------------------- | ----------------------------------------------- | ---------------------------- | ---------------- |
| **Type A: Ghost URLs** | 歷史重寫時改掉的 slug，但外部連結還指向舊 URL   | 部分（搜尋建議）             | 建立 redirect 表 |
| **Type B: 手寫錯誤**   | 讀者打錯字                                      | ✅ Smart 404 搜尋功能        | 現狀足夠         |
| **Type C: 跨語言缺口** | 讀者在 `/en/` 找中文文章但該文章沒翻譯          | ✅ Smart 404「尚未翻譯」提示 | 現狀足夠         |
| **Type D: 外部死連結** | Archive.org、社群分享、Wikipedia 引用指向舊 URL | 部分                         | 建立 redirect 表 |

**關鍵觀察**：Smart 404 解決了 Type B 和 C 的**讀者體驗**（讀者看到有用的建議），但沒解決 Type A 和 D 的**數量問題**（URL 繼續壞掉）。

### 對策三層

#### 🔴 P0 — 感知層（立即，5 分鐘能做）

在 `src/pages/404.astro` 加一段 GA4 event tracking script：

```javascript
<script is:inline>
  // Track 404 with the actual failed path
  if (typeof window !== 'undefined' && typeof gtag === 'function') {
    gtag('event', 'page_404', {
      failed_path: window.location.pathname,
      failed_url: window.location.href,
      referrer: document.referrer || '(direct)',
      language: document.documentElement.lang || 'unknown',
    });
  }
</script>
```

這會讓 GA4 的「自訂事件」裡出現 `page_404` 事件，包含：

- `failed_path`：哪個 URL 壞了
- `referrer`：從哪裡來的（判斷是 Google / 社群 / 直接輸入）
- `language`：讀者在哪個語言版本碰到 404

**1 週後可以導出「過去 7 天最常 404 的 top 20 URLs」**，然後才能做下一層決策。

#### 🟠 P1 — 修復層（需要 P0 的資料）

根據 P0 收集到的 top 404 URLs，建立一份 `public/_redirects`（Cloudflare Pages 原生支援）：

```
# Historical redirects (generated from 404 tracking)
/old-slug-1/          /new-slug-1/          301
/history/old-name/    /history/new-name/    301
```

還可以做更智能的：**自動偵測 slug 改名**。當一篇文章的 git history 顯示曾經有過不同的 filename，自動建立 redirect。

#### 🟡 P2 — 預防層（系統性）

- **Rename 保護**：改文章 slug 時，強制建立 redirect，不讓舊 URL 直接死掉
- **Sitemap diff**：每次 deploy 自動比對新舊 sitemap，任何「消失的 URL」要麼是故意刪除（報告出來確認），要麼是改名（自動建 redirect）
- **External backlink 監控**：periodically check 哪些外部站點連到我們，特別監控我們最常被連的 URL 是否還活著

### 這是「Smart 404」的盲區

Smart 404 是 UX 層的橋。這一層是感知層的橋。兩者缺一不可。**4/8 session δ 建了 UX 那一層，現在是補感知那一層的時候**。
